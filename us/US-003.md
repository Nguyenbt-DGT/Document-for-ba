## User Story

| Field | Value |
|---|---|
| **User story** | **As** a Manulife agent (FA) presenting a family insurance package, **I want** M-Smart to automatically fit the family package within my customer's stated budget — trimming coverage in a smart, transparent, rule-based way — and show the resulting premium per family member in a clear conversational format, **So that** I always receive a valid, presentable, budget-fit proposal without manually recalculating premiums myself. |

| **Post-condition:** | A **Priced, budget-fit Package** — every line item costed, trimmed if necessary, with a clear budget-fit indicator — is stored in session state and handed to US-004 for illustration PDF generation |

## 1. Scope

- **In scope:**
  - Deterministic, rule-based budget-fit trimming when the recommended package's total premium exceeds
    the stated budget, in a fixed priority order the FA can override.
  - Transparent before/after reporting of every trim decision.
  - Blocking package generation (with guidance) when the budget doesn't even cover the base product.
  - A two-tier "Ideal vs. Priority-First" presentation when the full package still can't fit.
  - Per-member premium calculation across up to 4 insured (MI + up to 3 OI).
  - Conversational (not tabular-only) premium display with a clear budget-fit indicator.
  - Re-calculation limited to the specific member/item the FA adjusts, without resetting the rest.
  - Friendly validation-error messaging and a response-time SLA.
- **Out of scope:**
  - Family intake, role resolution, or NLP parsing — belongs to **US-001**.
  - Selecting which products/riders are eligible or recommended — belongs to **US-002**.
  - Illustration PDF rendering, charts, or the printable document — belongs to **US-004**.
  - AI/ML-based (non-deterministic) budget optimization — must be rule-based.
  - Splitting the budget across multiple separate policies.
  - Annual premium increase projections and APE-based commission calculation.
  - Saving allocation preferences to the customer profile.

## 2. Requirement

#### R.01 Deterministic Budget-Fit Trimming Rule Engine

When the Recommended Package's total premium exceeds the stated budget, trim in this **fixed priority
order**, validating each step against the product/rider's minimum coverage rules before applying it:
1. Reduce the spouse rider's sum assured toward its minimum, then remove it if still over budget.
2. Reduce each child rider's sum assured toward its minimum, then remove it if still over budget
   (youngest child first if multiple children).
3. Last resort: reduce the base product's sum assured, never below the product's minimum.

#### R.02 Full-Fit (No Trim Needed) Confirmation

When the total premium already fits within budget, present the package unchanged with a clear
"✓ Trong ngân sách" indicator — no trimming logic runs.

#### R.03 Agent-Stated Priority Override

When the FA explicitly states a priority (e.g. *"Ưu tiên bảo vệ con, rider vợ không quan trọng"*), respect
it — trim in the stated order instead of the default — and confirm the adjusted order in the output.

#### R.04 Post-Trim Transparency & Surplus Handling

After trimming, show exactly what changed per member/item (original vs. adjusted sum assured, premium,
status). If the FA then requests a different arrangement, recalculate and apply it (warning, not blocking,
if the new total still exceeds budget). If a meaningful surplus remains after trimming, offer — but never
auto-apply — an increase to the base product or an added rider.

#### R.05 Budget Insufficient for Base Product — Hard Stop with Guidance

If the stated budget is below the minimum premium for any eligible base product, do not generate a
package — inform the FA of the minimum required and offer to adjust the budget or change the goal.

#### R.06 Budget-Stretch Two-Tier Presentation

When the full family package still cannot fit even after applying R.01, present two configurations: (1)
the Ideal configuration (full coverage, with the shortfall stated), and (2) a Priority-First fallback,
ranked base plan → spouse rider → older child → younger child. If the budget only covers the base plan,
output a single-insured pitch and mark every deferred member as *"Chưa đưa vào kế hoạch — có thể bổ sung
sau"*.

#### R.07 Multi-Member Premium Calculation

Call the premium calculation engine for every insured (MI + each OI) with the correct member identifier
and age, producing a per-member premium result for up to 4 insured in one session.

#### R.08 Conversational Premium Display Format

Render the priced package as a conversational message: one section per insured (role, applicable
riders, member premium), a footer with the family's total annual/monthly premium and budget comparison,
Vietnamese number formatting (thousands separated by a period), full Vietnamese rider names (with the
plan code in parentheses), and never an empty rider section for a member with no riders.

#### R.09 Re-calculation on Adjustment

When the FA requests a change to one member/rider, recalculate and update only that member's section —
every other member's coverage and premium is retained unchanged — then update the footer total and warn
the FA if the new total exceeds budget.

#### R.10 Validation Error Handling

When a requested sum assured or premium falls outside the valid range, show a friendly, specific message
(e.g. minimum sum assured for the product) — never expose a raw technical error code to the FA or
customer.

#### R.11 Performance SLA

Display the calculated result within 8 seconds for up to 4 insured, with a visible loading indicator while
waiting.

## 3. Business Rules & Validations

| Rule ID | Rule Name | Business Description | Impacted flow | Example Scenario |
|---|---|---|---|---|
| BR-US003-01 | Fixed trim priority order | Spouse rider → child rider (youngest first) → base product is the only valid default trim order. Refer to AC2 | update (pricing) | Package over budget by 12tr → spouse rider SA reduced first, base product untouched |
| BR-US003-02 | Trimming is rule-based, not AI-guessed | The same over-budget input must always produce the same trimmed output. Refer to AC2, AC3 | update (pricing) | Re-running the same over-budget scenario twice yields an identical trimmed package |
| BR-US003-03 | Every trim keeps the package valid | No line item may end up below its minimum coverage/premium; no rider may exist without a valid base product. Refer to AC3 | update (pricing) | Trimming never drops the base product's sum assured below its published minimum |
| BR-US003-04 | Agent priority overrides default order | An explicit FA priority statement changes the trim order for that session. Refer to AC4 | update (pricing) | "Ưu tiên bảo vệ con" → child rider protected, spouse rider trimmed first instead |
| BR-US003-05 | Surplus is offered, never auto-applied | Any leftover budget after trimming is proposed to the FA, who must explicitly accept it. Refer to AC7 | update (pricing) | 100.000đ surplus after optimization is mentioned, not silently spent on a bigger rider |
| BR-US003-06 | Post-trim override is advisory, not a hard block | The FA may accept an arrangement that still exceeds budget after an explicit override; the system warns but proceeds. Refer to AC6 | update (pricing) | FA removes spouse rider manually, keeps child rider even though total is still over budget — allowed with warning |
| BR-US003-07 | Auto-optimized package must meet the APE floor | Any auto-trimmed package must still satisfy the ≥10,000,000 VND APE minimum (per MGAFS-2033). Refer to AC12 | update (pricing) | An optimizer suggestion that would drop APE below 10tr is not offered as-is |
| BR-US003-08 | No empty rider sections | A member with zero riders never shows an empty rider block in the output (per MGAFS-1781). Refer to AC10 | read (display) | MI with no riders shows only the base product line, no blank "Riders:" heading |
| BR-US003-09 | Vietnamese number formatting | All monetary values use a period as the thousands separator (per MGAFS-2050). Refer to AC10 | read (display) | 45,500,000 đ is rendered as "45.500.000 đ" |
| BR-US003-10 | Adjustment retains unrelated members | Recalculating one member's coverage must not reset any other member's already-confirmed coverage (per MGAFS-1858, MGAFS-1865). Refer to AC11 | update (pricing) | Increasing MI's sum assured does not clear the already-confirmed spouse/child riders |
| BR-US003-11 | No raw technical errors shown to FA | Backend/API validation error codes must be translated into a friendly, actionable Vietnamese message (per MGAFS-2039). Refer to AC12 | exception (pricing) | A 4xx error from the pricing API becomes "STBH tối thiểu cho [sản phẩm] là 200.000.000 đ..." not an error code |

## 4. Impact and Risk

1. **Backend:** a deterministic rule engine for budget trimming, plus an aggregation layer calling the
   premium calculation engine once per insured (up to 4 calls) and assembling a conversational summary.
2. **API impact — the single biggest external risk to this story and the wider epic:** confirming whether
   `calculatePremiums` accepts a multi-insured payload in one call, or whether this story must fall back to
   sequential/parallel single-insured calls plus client-side aggregation (Pre-Sprint Blocker #1).
3. **Prerequisite bug risk:** MGAFS-1858 (M-Smart forgets riders on re-calculation) must be fixed **before**
   this story is built — a 3-OI family package would expose the bug three times worse than the existing
   single-persona case (Pre-Sprint Blocker #3).
4. **Regression risk:** existing single-persona premium calculation and budget-fit messaging must continue
   to work unchanged.

## 5. Non-Functional Requirements

1. **Performance:** premium results displayed within ≤ 8 seconds for up to 4 insured, with a loading
   indicator (*"⏳ Đang tính phí cho gia đình anh/chị..."*) while waiting.
2. **Determinism:** the trimming algorithm must produce the same output for the same input every time —
   explicitly not left to an LLM to "decide" case by case.
3. **Language & formatting:** Vietnamese-only messaging; Vietnamese thousands-separator number format.
4. **Audit:** every trim/adjustment action is retained in session history so re-calculation never silently
   loses prior confirmed state (supports BR-US003-10).

## 6. Dependencies / References

1. **US-002 (Product & Rider Recommendation)** — upstream, provides the Recommended Package to price.
2. `calculatePremiums` API — must support a multi-insured payload in one call, or this story implements
   sequential calls + aggregation as a fallback (Pre-Sprint Blocker #1, shared risk with US-002).
3. Bug **MGAFS-1858** (rider memory loss on re-calculation) — must be fixed as a prerequisite, not built
   in parallel.
4. Rider name mapping (plan code → full Vietnamese name) configuration table — Product/BA Team.
5. **MGAFS-2033** (APE minimum floor), **MGAFS-1781** (no empty rider sections), **MGAFS-2050** (VN number
   format), **MGAFS-2039** (no raw error codes) — existing standards this story must comply with, not
   reinvent.
6. Downstream: **US-004 (Illustration PDF Generation)** consumes the Priced Package produced here.

## 7. Technical Note

1. The trimming rule engine (R.01) should be implemented as a pure, unit-testable function independent of
   the conversational display layer (R.08) — this is what makes deterministic testing possible per
   BR-US003-02.
2. Re-calculation (R.09) must scope its API call/state update to the single changed member; do not
   re-invoke the full family pricing call unless every member's inputs actually changed.
3. Reuse the existing single-persona friendly-error-message mapping (R.10) rather than building a second
   parallel error-message table.

## 8. Acceptance Criteria

#### AC1. [Happy Path] Package already fits budget — no trimming applied

**Given** the Recommended Package's total premium (base product + all rider premiums) is ≤ the PO's stated budget  
**When** M-Smart evaluates the package  
**Then** the full package is presented unchanged with a **"✓ Trong ngân sách"** indicator  
**And** no trimming logic executes.

#### AC2. [Happy Path] Over-budget package trimmed in the fixed priority order

**Given** the Recommended Package's total premium exceeds the stated budget  
**When** M-Smart applies the trimming rule engine  
**Then** it reduces the spouse rider's sum assured toward minimum (then removes it) before touching any child rider, reduces child riders next (youngest first), and only as a last resort reduces the base product's sum assured, never below its minimum  
**And** it informs the FA which items were adjusted and why.

#### AC3. [Validation] Every trim keeps the package valid and consistent

**Given** any trimming action has been applied  
**When** M-Smart finalizes the trimmed package  
**Then** every remaining product and rider still satisfies its minimum coverage and premium rules  
**And** the package contains no orphaned rider without a valid base product.

#### AC4. [Edge Case] Agent-stated priority overrides the default trim order

**Given** the FA explicitly states a priority (e.g. *"Ưu tiên bảo vệ con, rider vợ không quan trọng"*)  
**When** the package needs trimming  
**Then** M-Smart trims the spouse rider before the child rider (reversing the default order)  
**And** the adjusted priority order is confirmed back to the FA in the output.

#### AC5. [Happy Path] Every change is reported transparently

**Given** M-Smart has trimmed the package to fit budget  
**When** the result is presented  
**Then** the output shows, per affected member/item: original value, adjusted value, resulting premium, and status (kept / reduced / dropped), plus the new grand total and its budget-fit status.

#### AC6. [Edge Case] Agent overrides the automatic allocation after trimming

**Given** M-Smart has automatically trimmed the package  
**When** the FA requests a different arrangement (e.g. *"Bỏ rider vợ/chồng, giữ nguyên rider con"*)  
**Then** M-Smart applies the override and recalculates  
**And** if the new total still exceeds budget, it warns the FA but allows them to confirm and proceed — this is advisory, not a hard block.

#### AC7. [Edge Case] Surplus budget after trimming is offered, not auto-applied

**Given** after trimming, remaining budget is greater than zero and large enough to meaningfully increase a rider's coverage  
**When** M-Smart finalizes the trimmed package  
**Then** it informs the FA of the surplus and offers to increase the base product or add a rider  
**And** it waits for the FA's explicit decision — it never auto-applies the increase.

#### AC8. [Negative Case] Budget too small even for the base product

**Given** the stated budget is below the minimum premium for every eligible base product  
**When** M-Smart evaluates the package  
**Then** it does **not** generate a package and informs the FA: *"Ngân sách X triệu/tháng chưa đủ cho sản phẩm cơ bản (tối thiểu Y triệu/tháng). Anh/chị có muốn điều chỉnh ngân sách hoặc thay đổi yêu cầu không?"*  
**And** offers explicit options: *[Điều chỉnh ngân sách]* / *[Đổi mục tiêu bảo hiểm]*.

#### AC9. [Edge Case] Full package still over budget after trimming — two-tier presentation

**Given** even the maximally-trimmed package (base plan only) does not fit within the budget for the full 4-person family, or fitting requires deferring members entirely  
**When** M-Smart finalizes the recommendation  
**Then** it presents two configurations: (1) Ideal — full coverage with the exact shortfall stated, and (2) Priority-First — ranked base plan → spouse rider → older child → younger child  
**And** asks the FA to confirm which configuration to proceed with, or offers to adjust the budget  
**And** if the budget only covers the base plan, it outputs a single-insured pitch and marks every deferred member *"Chưa đưa vào kế hoạch — có thể bổ sung sau"*.

#### AC10. [Happy Path] Multi-member premium calculated and displayed conversationally

**Given** a finalized (possibly trimmed) package for MI + up to 3 OI  
**When** M-Smart calculates and displays premiums  
**Then** it shows one section per insured (role/age/gender, applicable riders with full Vietnamese names and plan codes in parentheses, member premium), a footer with the family's total annual and monthly premium, and a clear budget-fit indicator (✅ within budget / ⚠️ over budget)  
**And** it never renders an empty rider section for a member with no riders, and uses Vietnamese thousands-separator number formatting throughout.

#### AC11. [Edge Case] Adjustment recalculates only the changed member

**Given** the FA requests a change to one member's coverage (e.g. *"Tăng STBH của NĐBH chính lên 600M"*)  
**When** M-Smart recalculates  
**Then** only that member's section is updated with the new premium — every other member's section retains its prior confirmed value unchanged  
**And** the footer total is updated, and the FA is warned if the new total exceeds budget.

#### AC12. [Exception Case] Invalid pricing request shows a friendly, actionable message

**Given** the FA requests a sum assured or premium outside the valid range for the product/rider  
**When** M-Smart processes the request  
**Then** it shows a specific, friendly Vietnamese message stating the actual valid minimum/maximum (e.g. *"STBH tối thiểu cho [sản phẩm] là 200.000.000 đ. Anh/chị muốn điều chỉnh lên mức tối thiểu không ạ?"*)  
**And** it never exposes a raw technical error code to the FA or customer.

#### AC13. [Integration Failure] Pricing engine unavailable, times out, or fails

**Given** the `calculatePremiums` call fails, times out, or is otherwise unavailable  
**When** M-Smart attempts to calculate the family's premiums  
**Then** it shows a friendly, retry-capable message rather than surfacing an unhandled failure  
**And** it does not silently return a partial, stale, or fabricated price to the FA while the failure is unresolved.
